# Önskelista UI Design Spec

**Date:** 2026-05-19
**Status:** Approved

## Overview

Frontend UI for Önskelista — a wishlist-sharing web app. Two user roles: Admin (creates/manages lists) and Guest (views lists, reserves gifts). Built with Next.js 14 Pages Router, shadcn/ui + Tailwind CSS, client-side Supabase fetching.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Styling | shadcn/ui + Tailwind CSS | Pre-built components (dialogs, forms, cards) accelerate development |
| Visual style | Warm & playful | Pastel tones, rounded corners — fits the gift/party theme |
| Architecture | Pages Router + client-side fetching | No SSR needed — guests arrive via direct links, not search engines |
| Reservation UX | Name-matching for unreserve | Simple, low-friction. App is for friends/family, not sensitive data |
| Dark mode | No — light only | Warm pastel palette works best in light mode, halves design work |

## Pages and Routing

```
/                       → Landing page (pitch + CTA)
/login                  → Login (Supabase Auth)
/signup                 → Registration
/admin                  → Dashboard — all my wishlists
/admin/list/[id]        → Edit wishlist + view reservations
/list/[url]             → Guest view — public wishlist (url = slug)
```

### Navigation

- **Guest:** No navbar. Wishlist content only. Discreet "Skapa din egen önskelista" link in footer.
- **Admin:** Simple header with "Mina listor" + logout button.

## Data Flow

### Supabase Queries Per View

| View | Query | Auth required |
|------|-------|--------------|
| Guest `/list/[url]` | `guest_wishes` view filtered on `wishlists.url` | No |
| Guest reserve | `supabase.rpc('reserve_wish', ...)` | No |
| Guest unreserve | `supabase.rpc('unreserve_wish', ...)` | No |
| Admin dashboard | `wishlists` filtered on `created_by = auth.uid()` | Yes |
| Admin list | `wishes` + `reservations` joined | Yes |
| Admin CRUD | Insert/update/delete on tables directly | Yes |

### State Management

- No global state library — each page fetches its own data
- `useState` + `useEffect` for data fetching
- Custom hooks: `useWishlist(url)` for guest view, `useAdminWishlist(id)` for admin
- Optimistic updates: mark card as "Paxad" immediately on reserve, roll back on RPC failure

### Auth Flow

- `supabase.auth.onAuthStateChange()` in an `AuthProvider` wrapping admin pages
- Redirect to `/login` if unauthenticated on `/admin/*`
- After login: redirect to `/admin`

## URL Slug Generation

- 8-character slug via `nanoid` (e.g., `xK9mQ2pL`)
- Generated client-side, stored in `wishlists.url`
- Displayed as `/list/xK9mQ2pL`
- UNIQUE constraint handles collisions — retry on conflict

## Error Handling

| Scenario | Behavior |
|----------|----------|
| Invalid slug | Friendly 404 page: "Listan hittades inte" |
| Reserve race condition | Toast: "Någon hann före!" |
| Network error | Toast with retry button |
| Auth error | Redirect to `/login` |

## Security

- Guest view uses `guest_wishes` database view — `reserved_by` and `guest_name` are never exposed to the client (AC-9)
- Reserve/unreserve operations go through `SECURITY DEFINER` database functions
- Admin pages protected by Supabase Auth + `AuthProvider` redirect
- RLS policies enforce that admins can only access their own lists
